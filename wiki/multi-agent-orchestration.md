# Multi-Agent Orchestration

## Summary

하나의 모델이 채팅 창 안에서 모든 일을 처리하도록 강제하는 대신, 역할별로 특화된 여러 에이전트에 작업을 분담시키고 오케스트레이터가 이를 조정하는 패턴이다. 각 역할에 가장 적합한 도구를 붙이면 단일 모델 구성보다 더 안정적이고 실용적인 개인 운영 시스템이 된다는 것이 핵심 주장이다.

## Details

전형적인 구성은 오케스트레이터 계층과 전문가 계층을 분리한다.

- **Orchestrator**: 메모리, 도구, 예약 작업(cron), 메시징을 보유하고 실제로 시스템에서 작업을 실행하는 항상 켜진 조정자. 직접 처리할지 전문가에게 위임할지 결정한다.
- **Main agent brain**: 일반적인 앞뒤 조정과 도구 사용을 담당하는 메인 추론 모델.
- **Specialist worker**: 코딩처럼 특정 작업에 집중된 전문가. 오케스트레이터가 제한된 작업을 전달하고 결과를 읽어 검증한다.
- **Interface / execution layer**: 원격 제어용 인터페이스(예: Telegram)와 파일·셸·이메일 등을 다루는 로컬 실행 환경.

워크플로우는 "사용자 요청 → 오케스트레이터가 위임 여부 판단 → 전문가가 작업 수행 → 오케스트레이터가 결과 검증·테스트·보고"의 루프로 돌아간다. 소스에 따르면 성능이 낮은 하드웨어에서 로컬 LLM 하나에 모든 것을 맡기는 방식(예: Ollama)은 전체 에이전트 사용에는 너무 느렸고, 역할 분담 방식이 더 잘 작동했다.

이 패턴의 실무적 장점 하나는 비용 구조다. 구독 기반 CLI([[claude-code|Claude Code]] 등)를 서브프로세스로 호출하면 토큰당 API 과금을 피하고 정액 구독 범위 안에서 전문가 모델을 활용할 수 있다. 다만 이 비용 이점은 제공업체 정책에 종속적이다(needs-update: 소스는 2026-06-15부터 `claude -p`/Agent SDK 사용이 Claude 구독 풀에서 분리된다는 정책 변경을 언급한다).
